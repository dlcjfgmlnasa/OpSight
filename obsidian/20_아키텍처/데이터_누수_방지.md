# 데이터 누수 방지 — 시뮬레이션 시간 t 이후 데이터는 절대 보지 않는다

> Project brief §13.2. 모든 tool 이 SimClock 의 `now_s` 를 인지하고, 그 이후 시점의 데이터를 읽으려 하면 거부한다.

## 데이터 누수가 뭐고 왜 막아야 하나

데이터 누수 (data leakage) 는 ML 평가에서 *시간 흐름을 어긴 정보* 를 사용하는 걸 말한다. 예를 들어:

> "10시 50분 시점에서의 저혈압 risk 예측" 작업을 하면서, 11시 05분의 ABP 값을 봤다.

이러면 모델이 *과대 평가* 된다. 학습 / 평가 결과는 매우 좋아 보이지만, 실제 배포 시에는 그 미래 정보를 못 볼테니까 성능이 폭락한다. 우리가 의도하는 "수술 중 실시간 예측" 의 시뮬레이션이라면 더더욱 치명적이다.

## VitalAgent 가 누수를 막는 방법 — 강한 invariant 한 줄

> 모든 tool 은 `SimClock.now_s` 를 인지한다. `query_window_end_s > clock.now_s` 인 query 는 거부한다.

이 invariant 가 시스템 어디서도 깨지지 않아야 한다. 코드는 두 곳에서 강제한다.

### 첫 번째 — `SimClock.assert_le`

```python
# vitalagent/sim_clock.py

class SimClock:
    def assert_le(self, query_window_end_s: float) -> None:
        if query_window_end_s > self._now_s:
            raise ValueError(
                f"data leakage: query_window_end_s={query_window_end_s} "
                f"exceeds current sim_time={self._now_s}"
            )
```

조용히 통과시키지 않고 `raise` 한다. Hard fail.

### 두 번째 — Tool wrapper 의 leakage guard

모든 EMR tool (그리고 FM tool 도 마찬가지) 이 `_leakage_guard` 를 호출한다. 위반 시 예외 대신 `ToolError` 를 반환해서 trace 에 기록된다.

```python
# vitalagent/tools/emr_tools_stub.py

def _leakage_guard(request, clock, query_window_end_s):
    if query_window_end_s > clock.now_s:
        return ToolResponse(
            error=ToolError(
                type="leakage_violation",
                message=f"query_window_end_s={query_window_end_s} exceeds clock.now_s={clock.now_s}",
            ),
            ...
        )
    return None
```

자세한 건 [[30_코드_워크스루/05_tools_layer]].

## 실제로 누수가 0건인지 확인하는 방법

```python
# tests/integration/test_smoke_single_case.py

def test_smoke_no_leakage_within_graph_run(...):
    # graph 를 끝까지 실행
    # ...
    events = read_trace(trace_path)
    tool_results = [e for e in events if e["event"] == "tool_result"]
    for e in tool_results:
        assert e["payload"]["ok"], f"unexpected tool failure: {e}"
```

300개 이상의 tool_result 를 모두 확인했을 때 leakage 0건이어야 PASS. 100 case e2e 도 마찬가지로 확인한다.

## sim_time 과 wall_time 은 다르다

쉽게 헷갈리는 부분이라 분리해서 적어둔다.

```
wall_time : 실제 사람 시계 시간 (지금 11시 30분이라면 11:30:00.123)
sim_time  : 시뮬레이션 안의 환자 시간 (case 시작 후 90.5초)
```

같은 wall_time 에 여러 case 를 동시에 시뮬레이션할 수 있다. 각 case 는 *자기 자신만의* sim_time 을 갖는다.

```python
clock = SimClock(start_s=0.0)
# ...
clock.tick(30.0)   # sim_time → 30.0
clock.tick(30.0)   # sim_time → 60.0
```

매 tick 마다 시뮬레이션 안에서 30초씩 진행된다. 실제 wall_time 은 (mock FM 일 때) ms 단위로 빠르다. 진짜 LLM 이 합류하면 wall_time 과 sim_time 이 비슷한 수준으로 가까워질 수 있다.

## Wall 이 sim 을 따라잡지 못하면 — 실시간성 손실

만약 sim_time 30초 tick 한 번 처리하는 데 wall_time 으로 35초가 걸린다면, "실시간 처리" 가 깨진다. 시간이 지날수록 점점 뒤쳐진다.

그래서 매 tick 의 wall 실측 시간을 함께 기록한다.

```python
@dataclass
class TickMeasurement:
    sim_time_before_s: float
    sim_time_after_s: float
    wall_start: float
    wall_end: float

    @property
    def wall_elapsed_s(self) -> float:
        return self.wall_end - self.wall_start
```

이 metric 으로 Shallow loop 의 15초 예산이 지켜지는지 추적한다 (project brief §6.1). 현재 mock FM + placeholder LLM 으로는 약 5ms, 진짜 Light LLM 합류 시 3–8초 예상 — 예산 안.

자세한 건 [[30_코드_워크스루/04_state_clock_triggers]].

## "실시간" 이 정확히 무슨 뜻인가

VitalAgent 의 "실시간" 은:

- **시뮬레이션된 실시간** (simulated real-time): 과거의 VitalDB recording 을 시간 순서대로 stream. 시점 t 에서는 t 이전의 데이터만 보인다.
- **Prospective real-time** (실제 환자에 붙어서 실시간) 은 PoC 범위 밖이다. NRF 도전형 2–3년차 후속 과제.

자세한 건 project brief §10.

## 어떤 패턴이 누수이고 어떤 패턴은 OK 인가

종종 혼동되는 경계 케이스들:

| 패턴 | 누수 발생 |
|------|-----------|
| Tool 9 `query_vasoactive_drugs` 호출 시 `time_window = (start, end)` 의 `end` 가 `clock.now_s` 를 초과 | ✅ 발생 — 즉시 `leakage_violation` 반환 |
| FM `predict_hypotension` 에 입력된 신호가 *미래 sample 을 포함* | ✅ 발생 (신호 ingest 단계에서 차단해야 함) |
| `forecast_signal` 이 *미래의 prediction* 을 반환 | ❌ OK — 미래 *예측* 은 정상, 금지되는 건 미래 *실측 데이터* |
| State 의 `risk_history` 에 *현재 tick 에서 막 추가한* sample | ❌ OK — 현재 tick 까지의 데이터는 정상 |

요약: **금지되는 건 미래의 실측 데이터**. 미래의 예측은 허용된다.

## Signal ingest 단계에서도 차단해야 한다

지금 코드는 *tool layer* 의 leakage 만 강제한다. 그런데 그 위에 있는 **신호 ingest 자체** 에서도 30초 window 일 때 그 window 안의 sample 만 가져와야 한다. 만약 ingest 가 미래 sample 을 슬쩍 끼워 넣으면 tool layer 가 못 막는다.

이 책임은 signal-ingest-engineer 영역이고, baseline 학습 / VitalDB exploration 작업에서 정밀화한다.
