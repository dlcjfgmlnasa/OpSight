# 40. 진행 상황 — 지금 어디까지 왔는가

> 이 문서는 **처음 보는 사람이 읽어도** 우리가 무엇을 만들었는지 알 수 있도록 풀어 썼다. 표·테스트 개수·sprint 로그 등 디테일은 문서 뒤쪽에 모아두었다.

## 한눈에

지금 OpSight는 **실제 임상 신호 없이도 처음부터 끝까지 굴러가는 프로토타입**이다. 다음 단계는 둘로 나뉜다.

- **회의 없이 가능한 작업** — 진짜 LLM (vLLM Llama) 연결, 실 cohort baseline 학습, EMR tool real 전환.
- **이형철 교수님 그룹 회의 결과를 기다리는 작업** — 코호트 정책 확정 (소아 / ASA=6), surgery context 정제, intervention response head 설계, "현재 상태 (Tier 0)" 설계.

아직 진짜 Foundation Model (FM)이 도착하지 않았기 때문에 **Mock FM**으로 자리를 잡아두었다. 진짜 FM이 오면 config 한 줄만 바꿔서 끼워 넣게 설계해두었다 (ADR-011).

---

## 만들어진 것을 풀어 쓰면

### 1. 신호를 읽고 30초 단위로 다듬는 부분

수술실 모니터에서 나오는 ABP/ECG/PPG/EEG 같은 신호는 초당 수백 Hz 단위의 raw waveform이다. 우리는 이걸 그대로 모델에 던지지 않고 **30초 window**로 잘라서 통계 feature (mean, std, SQI 등)를 뽑는다. 그 다음 매 30초마다 한 번씩 agent가 "지금 어떤가?"를 묻는 cycle을 시뮬레이션한다.

이 부분의 코드는 `opsight/sim_clock.py`와 baseline 모델용 feature extractor (`tests/test_baselines.py`에 정의)에 있다. VitalDB 채널과 sampling rate 카탈로그 (총 196개 track)는 `docs/vitaldb_catalog.md`에 정리해두었다.

### 2. 예측을 내놓는 부분 — Mock FM 3-tier

진짜 FM은 아직 학습 중이다 (Stage 2에서 도착 예정). 그동안 agent를 개발하려면 **FM 자리에 무엇이든 끼워 넣어야** 한다. 그래서 세 단계로 만들어두었다.

- **Tier 1 (Stub)** — 그냥 random 숫자를 뱉는다. "코드 흐름이 끊기지 않는가"를 확인할 때 쓴다.
- **Tier 2 (Rule-based)** — 8개의 임상 휴리스틱 룰로 그럴듯한 값을 만든다. 예를 들어 "MAP < 65이면 hypotension risk가 올라간다" 식. real하진 않지만 *반응성*은 있다 (실제 신호에 따라 값이 움직임).
- **Tier 3 (Light ML)** — 가벼운 baseline (Logistic Regression, MLP 등) 4종을 wrap해서 FM 자리에 끼운다. Tier 2보다 한 단계 더 *학습된* 동작을 보여준다.

세 tier 모두 `BiosignalFMInterface`라는 8-method 인터페이스를 따른다. 그래서 진짜 FM이 와도 같은 인터페이스만 구현하면 바로 swap된다. 자세한 이유는 [[20_아키텍처/Mock_FM_3_Tier_전략]] 참조.

코드: `opsight/fm/{interface,result_types,factory,mock_stub,mock_rule_based,mock_light_ml}.py`.

### 3. 21개의 도구 (Tool Suite)

Agent가 손과 발로 쓰는 함수들. **LLM이 직접 골라 부르지 않고** rule이 어떤 tool을 부를지 결정한다 (이유: 환각으로 중요한 도구를 빠뜨리면 brief가 잘못되므로). 21개의 분포는 이렇다.

- **FM-based 7개** — risk 예측, 신호 품질 평가, anomaly score 등. FM의 8 method 중 7개에 1:1 매핑.
- **EMR-based 5개** — 차트 / 약물 / 수액 조회. 현재 stub 상태.
- **Knowledge / Comparative 2개** — 유사 case 검색, intervention response 예측 (둘 다 ADR-013에 의존).
- **Auxiliary 2개** — 수술 맥락 (surgery context) hint, quality-weighted fusion.
- **Signal Access 5개** — 새로 추가된 layer (ADR-016). 신호 자체에 직접 접근해서 HRV, baseline drift 같은 derived metric을 계산. NeuroKit2 사용.

자세한 spec은 [[20_아키텍처/21_Tool_Suite]]. 코드는 `opsight/tools/*.py`에 흩어져 있고 envelope (`ToolRequest`/`ToolResponse`)은 모든 tool이 공유한다.

⚠️ **VitalDB 탐색 결과 (아래 "VitalDB 약물·수액 탐색" 절)** 때문에 **Tool 10 (`query_fluid_blood`)** 의 scope이 축소되었다. real-time stream이 불가능해서 case-end retrospective + 임상의 annotation에 의존해야 한다.

### 4. 두 가지 모드와 분기 규칙 — Shallow ↔ Deep

OpSight는 같은 graph에서 두 모드를 번갈아 돈다.

- **Shallow** — 매 30초마다 5개의 빠른 tool만 호출해서 한 문장 narration (예: "혈압 안정, 위험 낮음")을 만든다.
- **Deep** — 위험 신호가 잡히면 21개 전체를 호출해서 9-section 한글 brief를 생성한다.

언제 Shallow → Deep으로 넘어가는지는 **7개의 결정론적 (deterministic) rule + 60초 cooldown**으로 정한다. LLM이 결정하지 않는다 (이유는 [[20_아키텍처/Trigger_7_Rules]]).

코드: `opsight/triggers.py` + `opsight/nodes/{shallow_loop,deep_brief}.py` + `opsight/graph.py`.

100 case로 end-to-end 돌려본 결과: p95 latency 4.8ms, 100/100 case에서 deep brief가 한 번 이상 발화, hypotension_risk 발화 66회 + cross_modal_inconsistency 발화 234회 (총 300 deep fire, case당 평균 3회).

### 5. 결과를 한글로 풀어 쓰는 부분 — LLM prompt

Tool들이 숫자를 돌려주면 그걸 한국어 brief로 만들어주는 건 LLM이다. 현재는 **vLLM Llama**가 들어갈 자리에 placeholder template (`opsight/llm/placeholder.py`)이 있고, **system prompt v1 + v2**가 작성되어 있다 (`prompts/v1_*.md`, `prompts/v2_heavy_deep_brief.md`). v2는 21-tool source mapping + Tool 21 paraphrase 금지 enforcement + 영문 변형 포함.

진짜 LLM 연결이 다음 큰 작업이다 (아래 "남은 작업").

### 6. 코호트와 수술 맥락

- **코호트 (cohort)** — VitalDB 전체 6,388 case 중 너무 짧은 수술 등을 제외하면 약 **5,946 row**가 남는다. 이 manifest는 `scripts/build_cohort.py`가 만든다. 소아 / ASA=6 처리는 회의 후 확정 (그동안 default 적용).
- **수술 맥락 (surgery context)** — "general surgery × 진행기 (maintenance)" 같은 4×3 phase grid + 11개 optype별 hint를 yaml 파일 (`docs/surgery_context.yaml`)에 정리. Tool 15가 이걸 읽어서 LLM 추론에 힌트를 준다.

### 7. 안전장치 — 임상 사실 가드 + 데이터 누수 방지

- **임상 사실 가드** — agent가 임상 진단을 단정하지 않는다. 모든 단정에는 `[CLINICIAN-REVIEW: 이형철 교수님 그룹 검토 필요]` marker가 붙어야 한다.
- **데이터 누수 (leakage) 방지** — 시뮬레이션 시점 t에서 t 이후 데이터를 읽으면 `leakage_violation` error. 모든 EMR tool에 적용. 자세한 건 [[20_아키텍처/데이터_누수_방지]].

## 아직 안 한 것

### A. 회의 없이도 진행 가능

1. **진짜 LLM 연결** — placeholder template을 vLLM Llama로 교체. system prompt와 9-section brief generator는 이미 작성됨.
2. **Tier 0 (현재 상태) 설계** — ADR-014 Accepted 후 Tool 21 stub을 Tier 0 wrap으로 교체.
3. **EMR tool real 전환** — stub 상태인 5개를 VitalDB `intraop_*` 컬럼 매핑으로. 단, per-event timestamp가 없어 진가가 제한적 — prototype 단계에서는 stub으로 충분할 가능성.

### B. 이형철 교수님 그룹 회의 결과에 의존

회의 안건은 `docs/meetings/agenda_vital_group_review.md`. 핵심 결정 사항:

- **ADR-012 — Surgery-specific Downstream**: 수술 유형별 downstream head를 만들 것인가, 단일 head로 갈 것인가.
- **ADR-013 — Intervention Response**: "약물 X를 dose Y만큼 주면 trajectory가 어떻게 될까"를 어떻게 학습/예측할 것인가. 아래 VitalDB 탐색 결과로 인해 **v1 = vasoactive + anesthetic infusion만, bolus/fluid/transfusion은 v2 이연** 안이 유력.
- **ADR-014 — Tier 0 (Current State)**: "지금 이 환자가 어떤 상태인가"를 별도 head로 출력할 것인가.

회의 후 결정되면 코호트 정책 / surgery context hint / Tool 21 wrap이 정제된다.

## VitalDB 약물·수액 탐색 (2026-05-17)

`docs/findings/pump_drug_findings.md` 참조. 100-case sample (seed=20260517)로 확인한 결과:

- **`PUMP*` / `DRUG*` 같은 채널은 schema에 0 hit.** 모든 drug infusion은 `Orchestra/<DRUG>_<VAR>` 형식 (51 unique track).
- **Vasoactive infusion 가용률은 모두 5% 미만** — PHEN 2.0% · NEPI 1.4% · DOPA 0.5%. Tool 9 cohort가 작다.
- **`cases.csv` 의 `intraop_eph` 50.3% / `intraop_phe` 13.2%는 case-end 누적값**, per-event timestamp 없음 → Tool 10은 real-time 자동 감지 불가. Orchestra 채널에 ephedrine 자체가 부재.
- **Fluid / blood도 마찬가지** (`intraop_*` 누적만).
- Intervention head 학습 가능 stream events ≈ 11,000 (vasoactive 1,000 + anesthetic 10,000).
- → Paper narrative는 **mixed-initiative** framing (fully-automatic 불가).

영향:
- Tool 9/10 spec은 [[20_아키텍처/21_Tool_Suite]]에 반영 완료.
- 회의에서 임상의 검토가 필요한 항목: drug class 매핑 (PHEN=vasopressor, DOPA dose-dependent), ephedrine IV-push 가정, bolus segmentation rule, vasopressor equivalent dose 정의.

## 명명법 — `plan_1.x`가 뭐냐

`.plans/stage1_preparation/plan_1.{1..8}_*.md`는 Stage 1의 작업 단위 분해 문서다. 각 plan 파일은 한 묶음의 작업을 정의하고, 작업이 끝나면 `[x]` 체크로 표시한다. **이 문서를 외울 필요는 없다** — 어떤 capability가 어떤 plan에 속해 있는지 매핑은 다음과 같다.

| 의미 | 어디 |
|------|------|
| VitalDB 탐색 / 채널 카탈로그 | plan_1.1 |
| Mock FM Tier 1 (stub) | plan_1.1.5 |
| 코호트 정의 (manifest) | plan_1.2 |
| FM Interface Protocol + Factory | plan_1.2.5 |
| EMR tool (5개) | plan_1.3 |
| Signal Access tool (5개, ADR-016) | plan_1.3.5 |
| Baseline 모델 (LR/MLP 등 4종) | plan_1.4 |
| Surgery context (yaml hint) | plan_1.5 |
| System prompt v1 (한글 7종) | plan_1.6 |
| Mock FM Tier 2 (rule-based) | plan_1.6.5 |
| 21-tool spec + 코드 | plan_1.7 |
| Mock FM Tier 3 (Light ML) | plan_1.7.5 |
| Dual-mode skeleton (graph + state) | plan_1.8 |

→ 이 매핑은 참고용이고, 본 문서 본문에서는 capability 이름으로만 부른다.
